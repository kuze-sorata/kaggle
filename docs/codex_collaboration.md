# Codex との進め方

このドキュメントは、Kaggle 作業を Codex と進めるときの運用基準です。

## 基本方針

- Codex には、今どのフェーズにいるかを明示して依頼する
- 依頼は 1 サイクル 1 目的に寄せる
- 実装だけでなく、記録更新まで一緒に頼む

## notebook / src の役割分担

- `notebooks/` は試作・EDA・仮説検証の場所として使う
- notebook では 1 実験につき 1 変更を基本にする
- notebook は比較用の観察や軽い CV 確認まで行ってよい
- notebook で良かった案だけを `src/` に正式反映する
- 提出に使う正式版は notebook で閉じず、`src/` 側にそろえる

## `train.py` / `predict.py` / `features.py` の運用

- `train.py` はその時点の正式な学習フローを表す
- `predict.py` はその時点の正式な推論と提出生成フローを表す
- `features.py` は採用済み特徴量だけを置く場所として使う
- notebook で試しただけの特徴量は `features.py` に入れない
- `train.py` と `features.py` は、採用が決まった内容だけで上書きしていく
- 過去の試行錯誤は `experiments/`、notebook、git 履歴で追う

## 1 サイクルの基準

- `exp001` のような真の baseline は、生列に最小限の前処理だけを入れた比較基準にする
- EDA で面白い観察があっても、次の実験では 1 仮説だけを追加して CV 差分を見る
- 「何が効いたか分からない状態」を避けるため、複数変更を同時に正式採用しない
- 効いたと判断した変更だけを `src/train.py`、必要なら `src/features.py` に反映する

## 依頼の最小セット

- 対象コンペ
- 現在フェーズ
- 何を決めたいか
- 既知の制約
- 更新してほしいファイル

## 依頼テンプレート

```text
対象: competitions/<competition_name>
フェーズ: 問題理解 / ベースライン / CV設計 / EDA / 特徴量追加 / モデル比較 / 提出 / 実験記録
目的: いま決めたいことを一文で書く
制約: 使ってよいデータ、使いたくない手法、時間制約など
依頼: 何をしてほしいかを具体的に書く
```

## フェーズ別の頼み方

### 問題理解

- 依頼例
  - 「このコンペの目的、評価指標、提出形式、リーク候補を整理して」
- 期待する出力
  - README 要約
  - 初期リスク
  - 次に見るべきデータ観点

### ベースライン

- 依頼例
  - 「最小構成の学習・推論パイプラインを作って、提出まで通して」
- 期待する出力
  - 再実行可能なコード
  - 実行手順
  - 初回結果の記録

### CV 設計

- 依頼例
  - 「この問題に合う CV を決めて、なぜその分割にするか説明して」
- 期待する出力
  - 分割方式
  - 評価指標
  - 想定されるリークやズレ

### EDA

- 依頼例
  - 「仮説に直結する観点だけで EDA 項目を絞って」
- 期待する出力
  - 見る順番
  - 期待する発見
  - 次の仮説候補

### 特徴量追加

- 依頼例
  - 「改善仮説を 3 件出して、最も筋の良い 1 件だけ実装して」
- 期待する出力
  - 仮説
  - 実装
  - CV 差分

### モデル比較

- 依頼例
  - 「前処理を固定して、モデル差だけ比較できるようにして」
- 期待する出力
  - 比較表
  - 採用理由
  - 却下理由

### 提出

- 依頼例
  - 「この実験に対応する submission を作って、追跡できる形にして」
- 期待する出力
  - 提出ファイル
  - 対応実験番号
  - 残すべきメモ

### 実験記録

- 依頼例
  - 「今回の変更を `experiments/index.md` と `expXXX.md` に反映して」
- 期待する出力
  - 一覧更新
  - 詳細ログ更新
  - 次の打ち手の整理

## Codex に任せると良い作業

- ディレクトリ構成の整備
- 学習・推論コードの実装
- 実験ログの更新
- 比較表の整形
- TODO の更新

## 人が最終判断すべきこと

- どの仮説を優先するか
- LB の読み方
- 実験を打ち切るか続けるか
- コンペに固有のドメイン解釈

## セッションの終わりにやること

- `TODO.md` を更新する
- `experiments/index.md` を更新する
- 必要なら `workspace_notes/` に未整理メモを残す

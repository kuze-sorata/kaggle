# Kaggle の進め方

このドキュメントは、個別コンペではなく、`C:\Users\21td031\dev\kaggle` 全体で共通に使う標準ワークフローをまとめたものです。

## 目的

- 進め方を固定し、毎回のコンペで同じ型を反復できるようにする
- 実験を再現可能にし、CV と LB の関係を追えるようにする
- Codex に依頼するときの前提を揃え、手戻りを減らす
- 親エージェントと子エージェントの分担を固定し、並列実験でも競合しないようにする

## 標準フロー

1. 問題理解
2. ベースライン
3. CV 設計
4. EDA
5. 仮説ベースの特徴量追加
6. モデル比較
7. 提出
8. 実験記録
9. 区切り判断

## 開始時チェックリスト

新しいコンペを始めるときは、最初に次を揃える。

1. `competitions/<competition_name>/README.md`
2. `competitions/<competition_name>/TODO.md`
3. `competitions/<competition_name>/STATUS.md`
4. `competitions/<competition_name>/experiments/index.md`
5. `competitions/<competition_name>/requirements.txt`
6. `competitions/<competition_name>/notebooks/`
7. `competitions/<competition_name>/submissions/`
8. `competitions/<competition_name>/src/`

`.venv` はリポジトリルートに共通環境として作る。各コンペ配下には個別の `.venv` を作らない。

```powershell
cd C:\Users\21td031\dev\kaggle
python -m venv .venv
.\.venv\Scripts\Activate.ps1
pip install -r requirements.txt
```

## 親エージェント運用

- 同一コンペで複数実験を回す場合は、親エージェント 1 つと子エージェント複数の形を基本にする
- 親は `STATUS.md` で baseline、次の experiment id、active assignments を管理する
- 子は割り当て済みの `expXXX` だけを使い、原則として `notebooks/`、`experiments/`、`submissions/` の更新だけを担当する
- `src/` の正式更新、baseline の昇格、`experiments/index.md` の最終整合は親が担当する
- 会話履歴より `STATUS.md` を優先する

## 各フェーズの進め方

### 1. 問題理解

- 確認すること
  - 目的変数
  - 評価指標
  - 提出形式
  - テーブルの粒度
  - train/test の違い
- 成果物
  - コンペ README の要約
  - 最初に見るべきカラム一覧
  - リーク候補のメモ
- 完了条件
  - 何を当てるか、どう採点されるかを自分の言葉で説明できる

### 2. ベースライン

- やること
  - 最小前処理で学習から提出まで一度通す
  - 複雑さより再現性を優先する
- 成果物
  - 学習スクリプト
  - 推論スクリプト
  - 最初の提出ファイル
- 完了条件
  - ローカルで同じ手順を再実行できる

### 3. CV 設計

- やること
  - 問題に合った分割方法を決める
  - 評価指標を本番と揃える
  - seed と fold 数を固定する
  - 採用判断に使う差分のしきい値を意識する
- 典型例
  - 分類: `StratifiedKFold`
  - 回帰: `KFold`
  - グループあり: `GroupKFold`
  - 時系列: 時系列分割
- 完了条件
  - 以後の改善を CV 基準で比較できる

### CV 設計で最低限見ること

- fold ごとの score
- mean だけでなく std
- random_state を 1 個だけで過信しないこと
- Public LB とズレたときに、まず CV の不安定さを疑えること

小さい表形式コンペでは、`+0.002` 前後の差は split の偶然で消えることがある。  
Titanic のような小規模データでは、必要に応じて複数 seed や単純 holdout でも傾向を再確認する。

### 4. EDA

- やること
  - 欠損
  - 目的変数との関係
  - train/test の分布差
  - 明らかな外れ値や異常値
- 注意点
  - グラフを増やすこと自体を目的にしない
  - 仮説に繋がる観察だけを残す
- 完了条件
  - 次に試す特徴量や前処理の候補が言語化されている

### 5. 仮説ベースの特徴量追加

- 原則
  - 一度に一つずつ変える
  - 変更理由を先に書く
  - CV で差分を見る
- 変更対象の例
  - 新しい特徴量
  - 欠損補完の方法
  - カテゴリエンコード
  - 正規化や対数変換
- 完了条件
  - 何が効いて、何が効かなかったか説明できる

### 6. モデル比較

- やること
  - 前処理を固定してモデルだけ変える
  - モデルを固定して前処理だけ変える
  - 比較表を残す
- 注意点
  - アンサンブルは後半まで保留してよい
  - LB だけで良し悪しを決めない
- 完了条件
  - 現時点の第一候補モデルが決まっている

### モデル比較の原則

- まずは「特徴量を固定してモデルだけ変える」
- 次に「モデルを固定して特徴量だけ変える」
- 1 回の実験で、モデル変更と特徴量変更を同時に混ぜない
- CV で小幅改善しても、LB 悪化が出たら採用を保留する

### 7. 提出

- やること
- 提出ファイルを保存する
  - 対応する実験番号を残す
  - LB を記録する
- 管理方針
  - 提出ファイル名かログに実験番号を紐づける
  - Public LB の良し悪しで実験履歴を上書きしない
- 完了条件
  - どのコード・どの実験がどの提出に対応するか追える

### 8. 実験記録

- やること
  - `experiments/index.md` に一覧を追記する
  - `expXXX.md` に一次情報を書く
- 記録する内容
  - 目的
  - 仮説
  - 変更内容
  - CV
  - LB
  - 気づき
  - 次のアクション
- 完了条件
  - 数週間後に見返しても再現できる

### 9. 区切り判断

- やること
  - 現時点の正式 baseline を決める
  - 続ける理由があるか、次のコンペに移るべきかを判断する
  - README、TODO、STATUS に到達点を書く
- 典型的な区切りの条件
  - 小さい特徴追加で伸びなくなった
  - CV と LB のズレ要因を一度確認できた
  - 次の学びが別コンペの方で大きい
- 完了条件
  - 「このコンペで何が分かり、何を採用し、何を保留にしたか」が文書で残っている

## 1 サイクルの基本単位

- 1 仮説
- 1 変更
- 1 評価
- 1 記録

この粒度を守ると、改善の因果を追いやすいです。

## Codex に依頼するときの基本形

- 問題理解: 「このコンペの目的、評価指標、注意点を整理して」
- ベースライン: 「最小構成で学習から提出まで通るベースラインを作って」
- CV 設計: 「この問題設定に合う CV を提案し、理由も書いて」
- EDA: 「仮説につながる観点だけに絞って EDA 項目を整理して」
- 特徴量追加: 「仮説を 3 件出して、まず 1 件だけ実装して」
- モデル比較: 「比較条件を固定して、モデル差だけ見えるようにして」
- 提出: 「この実験に対応する提出ファイルを生成して記録して」
- 実験記録: 「今回の変更を `experiments/index.md` と `expXXX.md` に反映して」
- 区切り判断: 「このコンペの到達点を整理して、README と TODO に締めを残して」

## 避けること

- Public LB に合わせて手当たり次第に調整する
- 一度に複数の変更を混ぜる
- 実験を記録せず、後から思い出しで埋める
- notebooks だけで改善を進め、正式コードに戻さない
- CV の小差をそのまま採用判断に使う
- `STATUS.md` を更新せずに並列実験を始める

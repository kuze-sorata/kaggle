# exp001 - true baseline

## 日付

2026-05-22

## コンペ名

titanic

## フェーズ

ベースライン / 提出 / 実験記録

## 目的

最初の比較基準になる真のベースラインを作る。  
ローカル CV と Kaggle 提出が一通り通る状態を確認する。

## 仮説

生の主要列だけを使った最小構成の前処理と単純な線形モデルでも、Titanic の比較基準として十分なスコアが得られる。

## 理由

問題理解の直後は、複雑な特徴量設計よりも比較対象を早く作ることの方が重要だから。  
Titanic は小規模な表形式データなので、単純なベースラインでも改善の起点として機能しやすい。

## 変更内容

- `notebooks/exp001_baseline.ipynb` を真のベースライン用に作り直した
- 使用列を `Pclass`, `Sex`, `Age`, `SibSp`, `Parch`, `Fare`, `Embarked` の 7 列に限定した
- 数値列は中央値補完と標準化、カテゴリ列は最頻値補完と one-hot encoding に限定した
- `src/train.py` を同じ真のベースライン方針に合わせて修正した
- `LogisticRegression` を使った 5-fold `StratifiedKFold` のベースラインを評価した
- `submissions/baseline_submission.csv` を生成する構成に変更した

## 検証設計

- 分割方法: `StratifiedKFold(n_splits=5, shuffle=True, random_state=42)`
- 評価指標: Accuracy
- 比較条件: 生の 7 列だけを使う notebook / script 共通の真のベースライン。追加特徴量はまだ使わない

## 結果

- CV: mean `0.79687`, std `0.01457`
- CV fold scores: `0.78212`, `0.80337`, `0.79775`, `0.78090`, `0.82022`
- LB: `0.76555`

## 気づき

- 生の主要列だけでもローカル CV は 0.79 台に乗った
- 以前の `features.py` 依存の構成よりスコアは下がるが、比較基準としてはこちらの方が適切
- Public LB は以前の重い baseline と同じ `0.76555` だった
- この baseline を起点に `Title` や `HasCabin` などを 1 変更ずつ追加比較できる

## リスク

- 線形モデルなので、特徴間の非線形な関係は取り切れない
- Public LB だけで改善判断をすると過学習しやすい
- データサイズが小さいため、数千分の数の差は不安定な可能性がある

## 次のアクション

- `exp002` の EDA 結果を使って、`Title`、`HasCabin`、`FamilySize` などの追加候補を 1 件ずつ試す
- 次の実験では変更を 1 件に絞り、CV 差分が見えるようにする
- 真のベースラインに対して、どの特徴追加がどれだけ効いたかを記録する

# exp001 - notebook baseline

## 日付

2026-05-22

## コンペ名

titanic

## フェーズ

ベースライン / 提出 / 実験記録

## 目的

最初の比較基準になるベースラインを作る。  
ローカル CV と Kaggle 提出が一通り通る状態を確認する。

## 仮説

まずは最小構成の前処理と単純な線形モデルでも、Titanic の基準点として十分なスコアが得られる。

## 理由

問題理解の直後は、複雑な特徴量設計よりも比較対象を早く作ることの方が重要だから。  
Titanic は小規模な表形式データなので、単純なベースラインでも改善の起点として機能しやすい。

## 変更内容

- `notebooks/baseline_eda.ipynb` を作成した
- `train.csv` と `test.csv` を読み込み、基本的な欠損確認と主要列の集計を行った
- `src/features.py` の `TitanicFeatureEngineer` を notebook から利用した
- `src/preprocess.py` の前処理を notebook から利用した
- `LogisticRegression` を使った 5-fold `StratifiedKFold` のベースラインを評価した
- `submissions/notebook_baseline_submission.csv` を生成して提出した

## 検証設計

- 分割方法: `StratifiedKFold(n_splits=5, shuffle=True, random_state=42)`
- 評価指標: Accuracy
- 比較条件: notebook 上の最小ベースライン。モデルは `LogisticRegression`、前処理は既存の `features.py` と `preprocess.py`

## 結果

- CV: mean `0.82602`, std `0.01568`
- CV fold scores: `0.84358`, `0.81461`, `0.80337`, `0.82584`, `0.84270`
- LB: `0.76555`

## 気づき

- 最小構成でもローカル CV は 0.82 台に乗った
- Public LB は CV より低く、CV と LB の乖離を意識して改善を見る必要がある
- notebook から既存の `src` 部品を使う流れは問題なく回せた
- 提出まで CLI で一通り回せることを確認できた

## リスク

- `src/features.py` は既にかなり特徴量加工が入っており、厳密な意味では「素朴なベースライン」より一段進んだ構成になっている
- Public LB だけで改善判断をすると過学習しやすい
- データサイズが小さいため、数千分の数の差は不安定な可能性がある

## 次のアクション

- 次の実験では変更を 1 件に絞り、CV 差分が見えるようにする

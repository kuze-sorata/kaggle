# Titanic 問題設定の整理

## 概要

- コンペ名: Titanic - Machine Learning from Disaster
- タスク種別: 二値分類
- 目的: 各乗客が生存したかどうかを予測する

## 目的変数

- 目的変数: `Survived`
- ラベルの意味:
  - `0`: 生存しなかった
  - `1`: 生存した

## 評価指標

- 指標: `Accuracy`
- 意味:
  - 正解した予測の割合
  - 高いほど良い

## 提出形式

- 必須列:
  - `PassengerId`
  - `Survived`
- 期待される形:
  - `test.csv` の各乗客に対して 1 行ずつ予測を書く

例:

```csv
PassengerId,Survived
892,0
893,1
894,0
```

## データ構造

### `train.csv`

- 目的変数 `Survived` を含む
- 学習とローカル検証に使う

### `test.csv`

- `Survived` を含まない
- 予測と提出のために使う

## 先に確認したい主な列

- `PassengerId`: 行識別子
- `Pclass`: 客室クラス
- `Name`: 乗客名
- `Sex`: 性別
- `Age`: 年齢
- `SibSp`: 同乗している兄弟姉妹・配偶者数
- `Parch`: 同乗している親・子ども数
- `Ticket`: チケット識別子
- `Fare`: 料金
- `Cabin`: 客室情報
- `Embarked`: 乗船港

## 初期のモデリング上の含意

- `PassengerId` は基本的に識別子であり、そのまま使うかは慎重に判断する
- `Age`、`Cabin`、`Embarked` には欠損がある可能性が高い
- `Name`、`Ticket`、`Cabin` は生のままだと扱いづらいが、特徴量加工で有用になる可能性がある
- データサイズが小さいため、ローカル CV の設計が特に重要

## 注意点とリスク

- リーダーボード過学習:
  - Public LB は参考にはなるが、ローカル CV の代わりにはしない
- 前処理によるリーク:
  - 特徴量生成や欠損補完は、一貫した手順で適用する必要がある
- 小規模データによる不安定さ:
  - わずかなスコア差は有意でない可能性がある
